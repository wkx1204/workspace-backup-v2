### 案例 1：GitHub 发布认证失败

**原始问题**：无法推送到 GitHub

**发生时间**：2026-03-14  
**错误类型**：TECH-001 (API 使用错误)  
**状态**：✅ 已解决

#### 问题解决

**直接解决**：
- 多位置搜索 token 文件
- 使用 GitHub CLI 认证
- 手动创建仓库后推送

**1. 横向扩展（类似问题）**：
- ClawHub 发布未登录

```

│   ├── 有效性验证（发布前验证 token）
│   └── 备用方案（GitHub CLI, 手动创建）
├── 发布前检查

│   ├── 网络测试（ping github.com）
│   └── 文件检查（SKILL.md, README.md）
├── 错误处理
│   ├── 异常捕获（try-catch）
│   ├── 用户提示（友好错误信息）
│   └── 回滚机制（失败后恢复）
└── 日志记录
    ├── 详细日志（每个步骤）
    ├── 可追溯（时间戳/操作/结果）
    └── 便于调试（错误堆栈）
```

**3. 应用场景**：

## 🎯 举一反三方法论

### 1. 深度分析（5 Why）

**方法**：连续问 5 次"为什么"

**示例**：
- 问题：GitHub 推送失败
- Why 1：为什么失败？→ 认证失败
- Why 2：为什么认证失败？→ token 无效
- Why 3：为什么 token 无效？→ 位置不对
- Why 4：为什么位置不对？→ 没有标准
- Why 5：为什么没有标准？→ 缺乏规范

**根本原因**：缺乏凭证管理标准

---

### 2. 模式识别

**方法**：找到问题共性

**示例**：
- GitHub token 失败
- ClawHub token 失败
- 即梦 Cookie 失败

**共性**：认证凭证管理问题

**模式**：凭证搜索→验证→备用

---

### 3. 知识迁移

**方法**：应用到其他场景

**示例**：
- GitHub 认证方案
  ↓ 迁移
- ClawHub 认证
- 即梦登录
- 任何 API 认证

---

### 4. 系统优化

**方法**：从系统层面解决

**示例**：
- 问题：认证失败
- 系统方案：
  - 凭证管理标准
  - 发布前检查清单
  - 错误处理机制
  - 日志记录规范
